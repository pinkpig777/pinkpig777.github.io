# Week 1: Memory & Cache Locality

## The Mental Model: Hardware is the Bottleneck

> "The CPU is a Ferrari. Memory is a congested toll bridge."

In modern computing, the gap between CPU speed and memory latency (The "Memory Wall") is the primary hurdle in Quant Dev. A modern CPU can execute multiple instructions per cycle, but fetching data from main memory can take **hundreds of cycles**.

Performance is often not about instruction count. It is about:

> **How often my code forces the CPU to wait for data.**

If the CPU stalls waiting for data, it doesn’t matter how elegant the algorithm is. You aren’t writing code for an abstract machine; you are writing it for silicon.

---

# Core Terminology

## The Latency Gap
Modern systems use a tiered hierarchy to mask memory slowness. Each level trades capacity for speed.



* **L1 Cache**: Smallest, fastest (~0.5–1ns). Private to the core.
* **L2 Cache**: Mid-size, mid-speed (~3–10ns).
* **L3 Cache**: Large, slower (~10–40ns). Often shared across cores.
* **RAM**: Massive, but a performance killer (~60–100ns+).

## The Unit of Transfer: Cache Lines
Data doesn't move in bytes; it moves in **Cache Lines** (typically **64 bytes**).
* **Implication**: If you access a 1-byte `char`, the hardware fetches the surrounding 63 bytes anyway. Contiguous data means the next few elements are essentially "free."

## Locality: The Quant’s Best Friend
* **Spatial Locality**: Accessing memory addresses close to each other (e.g., walking an array).
* **Temporal Locality**: Re-using the same memory location quickly before it's evicted from cache.

## Pointer Chasing & Cache Misses
A **Cache Miss** occurs when the CPU looks for data in the cache and fails, forcing a fetch from a slower tier. 

Structures like **Linked Lists** are "pointer-heavy." Each node stores a pointer to a random heap location. The CPU prefetcher cannot predict where the next node is, leading to **Pointer Chasing**—a sequence of serial, high-latency memory fetches that stall the execution pipeline.

# Experiment: Two Queues, One Winner

To prove the impact of memory layout, I implemented two FIFO (First-In-First-Out) queues with identical $O(1)$ semantics but polar-opposite memory layouts.

### 1. ArrayQueue (The Ring Buffer)
This uses a contiguous block of memory.
* **Logic**: Uses a fixed-size array and bitmasking (`index & (capacity - 1)`) for wrap-around.
* **Hardware Fit**: High spatial locality. The CPU prefetcher sees the sequential access and pulls data into L1 before we even ask for it.

### 2. LinkedListQueue (The Performance Killer)
This uses traditional heap-allocated nodes.
* **Logic**: Each `push` calls `new`; each `pop` calls `delete`.
* **Hardware Fit**: Zero spatial locality. Nodes are scattered across the heap. The CPU must wait for each pointer to resolve before it can even *find* the next node.

# Benchmark Results

I ran the benchmarks on Apple Silicon using Google Benchmark in `Release` mode (`-O3 -march=native`).

| Implementation | Ops / Second | Latency (Approx) |
| :--- | :--- | :--- |
| **ArrayQueue** | ~2,700,000,000 | **~0.37 ns** |
| **LinkedListQueue** | ~70,000,000 | **~14.20 ns** |

> **The Verdict**: The ArrayQueue is **~39x faster** than the LinkedListQueue.

### Why the Gap is So Brutal:
1.  **Allocation Overhead**: `new` and `delete` involve the OS allocator, which often requires expensive locks.
2.  **Prefetcher Failure**: The prefetcher can't look ahead through a pointer; it can only look ahead through an address.
3.  **Instruction Density**: ArrayQueue uses cheap bitwise ANDs; LinkedListQueue involves complex heap management.

# The Real Lesson: Data-Oriented Design

I did not change the Big-O complexity. Both are $O(1)$. I did not find a "smarter" algorithm. 

**I optimized where the data lives.**

In trading systems, order books, message buffers, and matching engines must live in contiguous memory. If your data layout is chaotic, your latency will be non-deterministic, and your "fast" algorithm will be throttled by the hardware.
